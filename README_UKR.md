# Geocoder

## Опис завдання

Проект **Geocoder** — це сервіс геокодування, який повертає координати (широту й довготу) за текстовою адресою. Сервіс шукає адресу послідовно у трьох джерелах, і зупиняється на першому, де знаходить відповідь:

1. **Кеш Redis** — найшвидший варіант: якщо адресу вже шукали раніше, координати лежать у Redis і повертаються без звернення до бази чи до Google.
2. **База даних PostgreSQL** — якщо в кеші порожньо, сервіс шукає адресу в базі. Якщо знаходить, одразу кладе результат у кеш, щоб наступний запит уже не йшов у базу.
3. **Google Geocoding API** — якщо адреси немає ні в кеші, ні в базі, сервіс звертається до зовнішнього API Google. Знайдений результат зберігається і в базі даних, і в кеші, тож усі наступні запити за цією адресою обслуговуються миттєво.

Поле `source` у відповіді показує, звідки саме взято результат: `cache`, `database` чи `google`.

### Стійкість до збоїв

Сервіс спроєктований так, щоб часткові збої інфраструктури не приводили до помилки `500`, а деградували м'яко:

- Якщо **Redis недоступний**, читання й запис у кеш обгорнуті в обробку винятків: сервіс просто пише попередження в лог і продовжує працювати через базу даних та Google, а не падає.
- Якщо **Google API не відповідає через тайм-аут або обрив з'єднання** (тобто виникає тимчасова мережева помилка, а не помилка самого запиту), виклик автоматично повторюється до трьох разів, перш ніж сервіс визнає адресу нерозпізнаною. За повтори відповідає бібліотека Resilience4j.
- Якщо ж **Google повертає помилку статусу** (некоректний запит, вичерпана квота, помилка сервера), повтору не буде — таку помилку повторний запит однаково не виправить, тож не варто витрачати на нього квоту. Адреса просто позначається як нерозпізнана.
- **Порожня адреса** відсікається ще до звернення до Google: REST API одразу повертає код `400 Bad Request`, а у вебформі замість оманливого повідомлення «адресу не знайдено» показується чітке прохання ввести адресу.

## Основні технології

- **Java 21** — мінімальна версія, потрібна для роботи Spring Boot 4.
- **Spring Boot 4.1** — автоконфігурація, ін'єкція залежностей, тестові слайси (`@SpringBootTest`, `@WebMvcTest`).
- **Spring Data JPA** — робота з PostgreSQL через JPA-репозиторії.
- **Flyway** — версійовані SQL-міграції: схема бази даних і кілька прикладних адрес (Київ, Львів, Вінниця) створюються автоматично при старті застосунку.
- **Spring Data Redis** — кеш через `RedisCacheManager`, значення серіалізуються в JSON (`GenericJacksonJsonRedisSerializer`).
- **Spring RestClient** — клієнт для звернень до Google Geocoding API.
- **Resilience4j** (`resilience4j-spring-boot3`) — повторні спроби (retry) при тимчасових мережевих збоях у виклику Google.
- **Spring Boot Actuator** — ендпоінти стану сервісу (`health`, `info`).
- **Thymeleaf + HTMX** — вебінтерфейс без окремого JavaScript-фреймворку.
- **Testcontainers** — під час інтеграційних тестів піднімаються справжні Docker-контейнери PostgreSQL і Redis, а не заглушки.
- **JUnit 5, Mockito, AssertJ** — тестовий стек.

> **Примітка щодо назв залежностей у Spring Boot 4.** Проєкт використовує `resilience4j-spring-boot3`, бо артефакт `resilience4j-spring-boot4` ще не опубліковано в Maven Central (це відома проблема, її відстежують у [issue resilience4j#2427](https://github.com/resilience4j/resilience4j/issues/2427)); на практиці `resilience4j-spring-boot3` прекрасно працює поверх Spring Boot 4.1. Стартер AOP, потрібний для роботи анотації `@Retry`, у Spring Boot 4 називається `spring-boot-starter-aspectj`, а не `spring-boot-starter-aop`, як у третій версії — назву змінили розробники Spring Boot.

## Вебінтерфейс

Застосунок має простий інтерактивний вебінтерфейс на Thymeleaf та HTMX, доступний за адресою `http://localhost:8080`. У ньому можна ввести адресу, побачити її координати й джерело результату, а також переглянути розташування на вбудованій інтерактивній карті. Під формою пошуку сервіс показує останні 20 записів з бази даних і з кешу Redis — ці таблиці оновлюються після кожного пошуку. Якщо запит до бекенду завершується помилкою (наприклад, лежить база даних), угорі сторінки з'являється попереджувальний банер, а не мовчазна відсутність реакції на натискання кнопки.

## Стан сервісу та моніторинг

Через Spring Boot Actuator відкрито два ендпоінти, `health` та `info`:

- **`GET /actuator/health`** — загальний статус сервісу з деталями по компонентах `db` (PostgreSQL) і `redis`.
- **`GET /actuator/health/readiness`** — вужчий статус готовності, який враховує лише `db`. Кеш Redis необов'язковий, і сервіс продовжує працювати без нього, тож недоступність Redis не повинна змушувати оркестратор (наприклад, Kubernetes) знімати под з обслуговування трафіку. Водночас повний `/actuator/health` і далі показує компонент `redis` як `DOWN`, щоб проблема була видимою для моніторингу.

## Як запустити проект

1. Скопіюйте `.env.example` у `.env` і впишіть свій ключ Google Maps API у змінну `GOOGLE_API_KEY`.
2. Підніміть інфраструктуру (PostgreSQL на порту `5437`, Redis на порту `6379`):
   ```bash
   docker compose up -d
   ```
3. Запустіть застосунок:
   ```bash
   ./mvnw spring-boot:run
   ```

## Тести

```bash
./mvnw test
```

Перед запуском тестів переконайтеся, що Docker працює — інтеграційні тести піднімають реальні контейнери PostgreSQL і Redis через Testcontainers. Тестове покриття включає юніт-тести на Mockito, слайси `@WebMvcTest`, повноцінні інтеграційні тести `@SpringBootTest` проти справжніх контейнерів, а також окремі тести на сценарії деградації: недоступний Redis, помилка від Google, тайм-аут Google із подальшим повтором запиту.

## Структура проекту

```
src/main/java/com/example/geocoder/
├─ GeocoderApplication.java      # Точка входу, @EnableCaching
├─ CacheConfig.java              # Налаштування Redis-кешу (RedisCacheManager, JSON-серіалізація)
├─ CacheInspector.java           # Читання вмісту кешу для вебінтерфейсу
├─ RestClientConfig.java         # Бін RestClient для звернень до Google
├─ GeocodingService.java         # Бізнес-логіка: кеш → база → Google, з обробкою збоїв
├─ GeoLocation.java              # JPA-сутність запису адреси
├─ GeoLocationRepository.java    # JPA-репозиторій
├─ GoogleGeocodingClient.java    # Клієнт Google Geocoding API з повторними спробами (@Retry)
├─ GeocodingResult.java          # DTO результату геокодування
├─ GeocodingController.java      # REST API (/api/geocode)
└─ GeocoderWebController.java    # Вебконтролер (сторінка, HTMX-фрагменти)

src/main/resources/
├─ application.properties        # Конфігурація БД, Redis, Actuator, Resilience4j
├─ db/migration/                 # SQL-міграції Flyway
└─ templates/                    # Thymeleaf-шаблони (сторінка, фрагменти карти й таблиць)

src/test/java/com/example/geocoder/
├─ GeocodingControllerTest.java        # Тести REST-контролера (@WebMvcTest)
├─ GeocodingServiceTest.java           # Юніт-тести сервісу, зокрема сценарій недоступного кешу
├─ GeocodingIntegrationTest.java       # Інтеграційні тести з Testcontainers
├─ GoogleGeocodingClientTest.java      # Тести клієнта Google (успіх, помилка, порожній результат)
├─ GoogleGeocodingClientRetryTest.java # Тести повторних спроб при тимчасових мережевих збоях
├─ ActuatorHealthTest.java             # Тести ендпоінтів /actuator/health
└─ TestcontainersConfiguration.java    # Конфігурація контейнерів PostgreSQL і Redis для тестів
```
