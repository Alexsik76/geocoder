# Geocoder

## Опис завдання

Проект **Geocoder** — це простий сервіс для геокодування, який надає можливість отримати координати (широта, довгота) за заданою адресою. Сервіс використовує три джерела даних у пріоритетному порядку:

1. **Кеш** – найшвидший шлях, дані зберігаються у пам'яті за допомогою Spring Cache (`ConcurrentMapCacheManager`).
2. **База даних** – якщо дані відсутні в кеші, сервіс намагається знайти їх у PostgreSQL.
3. **Google Geocoding API** – у випадку, коли інформація не знайдена в кеші та БД, здійснюється запит до зовнішнього сервісу (мокований у тестах).

Після успішного отримання результату сервіс записує його в кеш і (у випадку запиту до Google) зберігає у базі даних. Це гарантує, що наступні запити за тією ж адресою будуть оброблені миттєво з кешу.

## Основні технології

- **Java 21** – мінімальна версія, підтримка актуальних мовних можливостей.
- **Spring Boot 3** – автоконфігурація, DI, тестові слайси (`@SpringBootTest`, `@WebMvcTest`).
- **Spring Data JPA** – робота з PostgreSQL, автоматичне створення репозиторіїв.
- **Spring Cache** – `@EnableCaching`, `ConcurrentMapCacheManager` для простого in‑memory кешу.
- **Testcontainers** – під час інтеграційних тестів запускаються реальні Docker‑контейнері PostgreSQL та Redis.
- **Mockito** – мокання зовнішніх залежностей (Google API, CacheManager) у юніт‑тестах.
- **Maven** – збірка проекту, плагін `maven-surefire-plugin` для запуску тестів.
- **JUnit 5** – тестовий фреймворк, включає `@Parallel` та `@Execution` для швидкого виконання.

## Архітектурні рішення

| Рішення | Обґрунтування |
|---------|----------------|
| **Конструкторна ін’єкція** для всіх залежностей (`GeoLocationRepository`, `GoogleGeocodingClient`, `CacheManager`) | Підвищує читабельність та тестованість, уникає проблем з полями, позначеними `@Autowired` у тестових контекстах. |
| **Ручна робота з кешем** замість `@Cacheable` | Дозволяє явно контролювати пріоритети (кеш > БД > Google) та уникнути конфліктів під час тестування `@WebMvcTest`, де кеш‑bean може бути відсутній. |
| **CacheManager як optional dependency** | У тестах, де кеш не потрібен, можна передати `null`, що спрощує створення `GeocodingService` без зайвих моків. |
| **Тестовий контекст із `@MockitoBean`** | Дозволяє підміняти `GoogleGeocodingClient` та `CacheManager` у `@SpringBootTest`, забезпечуючи ізольоване тестування логіки сервісу. |
| **Testcontainers** для інтеграційних тестів | Гарантує, що база даних та Redis запускаються у реальному середовищі, що збільшує достовірність тестів і виключає «залипання» локальних даних. |
| **Видалення запису з БД у тестах** перед другим викликом | Підтверджує, що результат дійсно береться з кешу, а не з бази. |
| **Використання `Optional`** у репозиторії та клієнті Google | Акуратно обробляє відсутність даних без `null`‑пойнтерів, підвищуючи безпечність коду. |

## Як запустити проект

```bash
# У Windows PowerShell (в корені репозиторію)
./mvnw clean test          # Запускає всі юніт‑ та інтеграційні тести
./mvnw spring-boot:run    # Запуск веб‑додатку (порт 8080 за замовчуванням)
```

## Структура проекту

```
src/main/java/com/example/geocoder/
├─ GeocoderApplication.java      # Основний клас, @EnableCaching
├─ CacheConfig.java              # Налаштування кешу (ConcurrentMapCacheManager)
├─ GeocodingService.java         # Бізнес‑логіка, ручна робота з кешем
├─ GeoLocationRepository.java    # JPA репозиторій
├─ GoogleGeocodingClient.java    # Обгортка над зовнішнім API (мок у тестах)
├─ GeocodingResult.java          # DTO результату геокодування
└─ ...

src/test/java/com/example/geocoder/
├─ GeocodingControllerTest.java # Тести контролера, мок CacheManager
├─ GeocodingServiceTest.java    # Юніт‑тести сервісу без кешу
├─ GeocodingIntegrationTest.java# Інтеграційні тести з Testcontainers
└─ TestcontainersConfiguration.java # Конфігурація Testcontainers
```
